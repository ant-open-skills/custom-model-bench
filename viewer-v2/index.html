The Bench

design doc · v2

Who this is for

Founders and engineers deciding which model to ship for a specific feature. The leaderboard answers "which one wins for my trade-off?" — the use-case pills re-weight the fit score so speed-critical and cost-sensitive teams see different rankings on the same data.

Fit score

A 0–100 composite per candidate. Each raw aggregate metric normalizes to a 0–100 sub-score (log-ish for latency and cost, linear on the success-rate tail), then weighted-averaged per use case. Weights sum to 1 and live in fit.js; a quality slot is reserved at weight 0 for rubric-graded results once they ship.

Components

  • <leaderboard> — sortable table with use-case pills, 4 summary cards, provider/tier chip filters
  • <model-label> — provider dot + serif display name + tier badge + mono model-id footer
  • <fit-bar> — 0–100 bar colored by tier; composite visible at a glance
  • <candidate> — per-candidate drill-down card with serif response body and mono metric footer
  • <tweaks-drawer> — density / accent / theme; bottom-right, persistent

What's deliberately cut

Radar chart over per-dimension scores — needs rubric-graded quality dimensions that don't exist in the data yet. Returns in Phase E.

Recent runs feed — no historical store yet. The Runs tab shows an empty state with the exact rebuild command.

Compare tray — overlaying two candidates requires radar, which see above.

Data

All content is sourced from comparison_*.json files committed in the repo. build-data.ts walks the five example directories and emits data.js as window.__BENCH. See MIGRATION.md for the full design-field → schema-field map.
